## 1. Description Writing Patterns

The description is the skill's only trigger mechanism. Claude sees only the name + description from the `available_skills` list and decides whether to use the skill.

### Understanding the Trigger Mechanism

Claude tends not to invoke skills for simple tasks it can easily handle with its built-in tools. A simple request like "read this PDF" might not trigger a skill even if the description is perfect. Complex, multi-step, specialized tasks have a higher skill trigger probability.

### Writing Principles

1. Describe **what the skill does** + **specific trigger situations** together.
2. Specify boundary conditions that distinguish similar but non-triggering cases.
3. Write slightly "pushy" -- compensate for Claude's tendency to be conservative about triggering.

### Good Examples

```yaml
description: "Read PDF files, extract text/tables, merge, split, rotate, watermark,
  encrypt/decrypt, OCR, and all other PDF operations. Use this skill whenever
  a .pdf file is mentioned or a PDF output is requested. Especially useful
  when transformation/editing/analysis is needed, not just 'read the PDF'."
```

```yaml
description: "All spreadsheet operations including adding columns, formula
  calculations, formatting, charts, and data cleaning for Excel/CSV/TSV files.
  When the user mentions a spreadsheet file -- even casually ('that xlsx in
  my downloads folder') -- use this skill."
```

### Bad Examples

- `"A skill that processes data"` -- too vague, unclear which files/operations
- `"PDF-related tasks"` -- no specific actions listed, no trigger situations described

## 2. Body Writing Style

### Why-First Principle

LLMs judge edge cases correctly when they understand the rationale. Providing context is more effective than coercive rules.

**Bad:**
```markdown
ALWAYS use pdfplumber for table extraction. NEVER use PyPDF2 for tables.
```

**Good:**
```markdown
Use pdfplumber for table extraction. PyPDF2 specializes in text extraction
and does not preserve table row/column structure. pdfplumber recognizes cell
boundaries and returns structured data.
```

### Generalization Principle

When issues are found in feedback or testing, generalize at the principle level rather than making narrow fixes for specific examples.

**Overfitting fix:**
```markdown
If there's a "Q4 Revenue" column, convert it to numeric.
```

**Generalized fix:**
```markdown
When a column name contains keywords implying numeric values (revenue, amount,
quantity, etc.), convert that column to numeric type. On conversion failure,
retain the original value.
```

### Imperative Voice

Use directive form: "Do X", "Use Y", "Generate Z". Skills are instructions, not descriptions.

### Context Conservation

The context window is a shared resource. Ask whether every sentence justifies its token cost:
- "Does Claude already know this?" -> delete
- "Would Claude make mistakes without this explanation?" -> keep
- "Is one concrete example more effective than a long explanation?" -> replace with example

## 3. Output Format Definition Patterns

Use when the artifact's format matters:

```markdown
## Report Structure
Follow this template exactly:

# [Title]
## Summary
## Key Findings
## Recommendations
```

Keep format definitions concise; include actual examples for greater effectiveness.

## 4. Example Writing Patterns

Examples are more effective than long explanations:

```markdown
## Commit Message Format

**Example 1:**
Input: Add JWT token-based user authentication
Output: feat(auth): implement JWT-based authentication

**Example 2:**
Input: Fix bug where password visibility toggle on login page doesn't work
Output: fix(login): fix password visibility toggle button behavior
```

## 5. Progressive Disclosure Patterns

### Pattern 1: Domain-based Separation

```
bigquery-skill/
  SKILL.md          (overview + domain selection guide)
  references/
    finance.md      (revenue, billing metrics)
    sales.md        (opportunities, pipeline)
    product.md      (API usage, features)
```

When the user asks about revenue, only finance.md is loaded.

### Pattern 2: Conditional Detail

```markdown
# DOCX Processing

## Document Generation
Generate new documents with docx-js. -> See [DOCX-JS.md](references/docx-js.md).

## Document Editing
For simple edits, modify XML directly.
**If tracked changes are needed**: See [REDLINING.md](references/redlining.md).
```

### Pattern 3: Large Reference File Structure

Reference files over 300 lines should include a table of contents at the top:

```markdown
# API Reference

## Table of Contents
1. [Authentication](#authentication)
2. [Endpoint List](#endpoint-list)
3. [Error Codes](#error-codes)
4. [Rate Limiting](#rate-limiting)

---

## Authentication
...
```

## 6. Script Bundling Criteria

Observe agent transcripts during test execution. Bundle when these patterns appear:

| Signal | Action |
|--------|--------|
| All 3 tests generate the same helper script | Bundle in `scripts/` |
| Same pip install/npm install every time | Specify dependency installation step in the skill |
| Same multi-step approach repeated | Document as standard procedure in the skill body |
| Same error followed by the same workaround every time | Document known issues and solutions in the skill |

Bundled scripts must undergo execution testing.

## 7. Data Schema Standards

Standard schemas for consistency in inter-skill data exchange. Usable for testing and evaluating skills generated by the harness.

### eval_metadata.json

Metadata for each test case:

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "User's task prompt",
  "assertions": [
    "The output contains X",
    "A file was generated in Y format"
  ]
}
```

### grading.json

Assertion-based grading results:

```json
{
  "expectations": [
    {
      "text": "Output contains 'Seoul'",
      "passed": true,
      "evidence": "Found 'Seoul area data extraction' in step 3"
    }
  ],
  "summary": {
    "passed": 2,
    "failed": 1,
    "total": 3,
    "pass_rate": 0.67
  }
}
```

**Field name note:** Use exactly `text`, `passed`, `evidence`. Do not use variants like `name`/`met`/`details`.

### timing.json

Execution time/token measurement:

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

Save `total_tokens` and `duration_ms` immediately from the sub-agent completion notification. This data is only accessible at notification time and cannot be recovered later.

## 8. What NOT to Include in Skills

- README.md, CHANGELOG.md, INSTALLATION_GUIDE.md, or other supplementary documents
- Meta information about the skill creation process (test results, iteration history)
- User-facing documentation (skills are instructions for AI agents)
- General knowledge that Claude already possesses
