# Risk Record Schema

YAML schema for `docs/.index/risk-file.yaml` produced by the `risk-control` skill. This
file is the single source of truth for the risk-file shape, the per-record field semantics,
and the validation rules the `validate` subcommand enforces. Auditors and downstream
consumers (the `traceability` matrix, the `evidence-pack` release packager) should treat
this file as the contract.

> **Loading**: Loaded under `tier: standard` and `tier: deep` via the skill's `ref_docs.schema`
> entry. Skip when invoking under `tier: light`.

## Top-Level Document Layout

```yaml
# docs/.index/risk-file.yaml
# Generated by /risk-control -- do not edit by hand
_meta:
  schema: "1.0.0"           # Schema version of this document
  generated: "YYYY-MM-DDTHH:MM:SSZ"  # ISO 8601 UTC timestamp
  generator: "risk-control"          # Skill name that produced this artifact
  matrix_source: "default"  # "default" or "risk-matrix.yaml" (per-project override)
  records: N                # Total number of risk records below

records:
  - <record>                # See "Record Schema" below
  - <record>
  - ...
```

`_meta.schema` follows semantic versioning. Consumers must check `_meta.schema` before
parsing records; mismatched majors mean the schema has changed in a backward-incompatible
way. The skill writes the matching schema version on every regeneration.

Records are sorted by `id` (ASCII order) on every write so the file diffs cleanly in
version control.

## Record Schema

Every record is one hazard or one risk-controlled scenario. Hazards (`H-NN`) describe the
underlying source of harm; risks (`R-NN`) describe one specific risk-controlled scenario
arising from a hazard. A simple project may use only `H-NN` records; a project that needs
to track multiple control variants per hazard uses `R-NN` records that reference the
parent `H-NN` via `hazard`.

```yaml
- id: R-01                              # MANDATORY -- primary key
  hazard: "Unintended therapy delivery" # MANDATORY -- one-line hazard description
  sequence_of_events: |                 # MANDATORY -- multi-line; the chain of events leading to harm
    1. Operator selects pre-set protocol
    2. Embedded validation skips a parameter check
    3. Device delivers therapy outside approved range
  harm: "Patient receives over-dose"    # MANDATORY -- one-line harm description
  severity: critical                    # MANDATORY -- enum, see "Enumerations" below
  probability: occasional               # MANDATORY -- enum, see "Enumerations" below
  P1: occasional                        # OPTIONAL -- ISO-14971 P1 (probability of hazardous situation)
  P2: probable                          # OPTIONAL -- ISO-14971 P2 (probability harm follows the situation)
  initial_risk: unacceptable            # MANDATORY -- derived from severity x probability via the matrix
  control_measures:                     # MANDATORY (non-empty) -- one or more risk-control measures
    - id: CM-01
      description: "Add range validation in firmware"
      priority: 1                       # ISO-14971-7.1 priority: 1=design, 2=protective, 3=information
      verification:                     # MANDATORY (non-empty per measure) -- verification activities
        - "TC-CALC-014: range-validation unit test"
        - "TC-INT-007: end-to-end protocol limits test"
  residual_severity: critical           # MANDATORY -- severity after control measures applied
  residual_probability: remote          # MANDATORY -- probability after control measures applied
  residual_risk: ALARP                  # MANDATORY -- derived from residual_severity x residual_probability
  requirements:                         # MANDATORY (may be empty) -- SRS ids the risk depends on
    - SRS-CALC-014
    - SRS-CALC-022
  verification:                         # OPTIONAL -- top-level verification activities (cross-references TC-)
    - TC-CALC-014
    - TC-CALC-022
  clause_refs:                          # MANDATORY (may be empty) -- ISO 14971 clause ids
    - ISO-14971-5.4
    - ISO-14971-7.1
    - ISO-14971-7.3
  status: controlled                    # MANDATORY -- enum, see "Status Field" below
  notes: ""                             # OPTIONAL -- free-form, single line, factual
```

### Field Definitions

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | Yes | Format `R-NN` (risk-controlled scenario) or `H-NN` (hazard). Two-digit sequence; `H-01`, `R-01`, `R-02`. Primary key for the record. |
| `hazard` | string | Yes | One-line hazard description (paraphrased from ISO-14971-5.4 hazard catalogue). |
| `sequence_of_events` | string (multi-line) | Yes | Chain of events leading from the hazard to the harm. ISO-14971-5.5 evidence. |
| `harm` | string | Yes | One-line description of the harm to the patient/operator/environment. |
| `severity` | enum | Yes | Severity of the harm before control measures. See "Enumerations" below. |
| `probability` | enum | Yes | Probability of the harm occurring before control measures. See "Enumerations" below. |
| `P1` | enum | No | ISO-14971 P1: probability of the hazardous situation occurring. Recorded when the project uses the P1/P2 decomposition; otherwise omitted. |
| `P2` | enum | No | ISO-14971 P2: probability that, given the hazardous situation, harm follows. Recorded when the project uses the P1/P2 decomposition; otherwise omitted. |
| `initial_risk` | enum | Yes | Risk level before control measures, derived from severity x probability via the matrix. One of `acceptable`, `ALARP`, `unacceptable`. The skill computes this; operators do not set it directly. |
| `control_measures` | list[object] | Yes (non-empty) | One or more risk-control measures. See "Control Measure Schema" below. |
| `residual_severity` | enum | Yes | Severity after control measures applied. Often equal to `severity` (controls usually reduce probability, not severity). |
| `residual_probability` | enum | Yes | Probability after control measures applied. |
| `residual_risk` | enum | Yes | Risk level after control measures, derived. ISO-14971-7.3 requires this be evaluated. |
| `requirements` | list[string] | Yes (may be empty) | SRS ids the risk depends on for control. Format `SRS-{CAT}-{NNN}` per `traceability/reference/matrix-schema.md`. Each must resolve in `docs/.index/manifest.yaml`. |
| `verification` | list[string] | No | Top-level verification cross-references (TC ids). Most projects keep verification on each `control_measure` instead; this top-level field is provided for compatibility with simple risk files. |
| `clause_refs` | list[string] | Yes (may be empty) | ISO 14971 clause ids the record bears on. Format per `traceability/reference/matrix-schema.md` clause format section. |
| `status` | enum | Yes | One of `draft`, `controlled`, `accepted`, `transferred`, `deferred`. See "Status Field" below. |
| `notes` | string | No | Free-form, single line. Auditors read this; keep it factual. |

### Control Measure Schema

Each entry under `control_measures` is one risk-control measure plus its verification.

```yaml
- id: CM-01                             # MANDATORY -- format CM-NN, unique within the parent record
  description: "Add range validation in firmware"  # MANDATORY -- one line
  priority: 1                           # MANDATORY -- 1=design, 2=protective, 3=information
  verification:                         # MANDATORY (non-empty) -- verification activities
    - "TC-CALC-014: range-validation unit test"
    - "TC-INT-007: end-to-end protocol limits test"
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | Yes | Format `CM-NN`. Unique within the parent record. |
| `description` | string | Yes | One-line description of the control measure. |
| `priority` | integer | Yes | ISO-14971-7.1 risk-control option priority. `1`=inherent safety by design, `2`=protective measures, `3`=information for safety. Lower is better; `validate` warns when a record's lowest priority is `3` (information-only) without a justification in `notes`. |
| `verification` | list[string] | Yes (non-empty) | Verification activities that demonstrate the control is effective. Plain strings; typically `TC-{CAT}-{NNN}` test ids with optional inline title. |

## Identifier Format Conventions

| Entity | Format | Example | Note |
|--------|--------|---------|------|
| Hazard | `H-NN` | `H-03` | Two-digit sequence; matches `router.yaml` `id_routes.H` per `traceability/reference/matrix-schema.md`. |
| Risk-controlled scenario | `R-NN` | `R-12` | Two-digit sequence; project-private. Risks reference their parent hazard via the `hazard` field's prose. |
| Control measure | `CM-NN` | `CM-01` | Two-digit sequence; unique within a single record. |
| Requirement | `SRS-{CAT}-{NNN}` | `SRS-CALC-014` | Same format as the traceability matrix. |
| Test case | `TC-{CAT}-{NNN}` | `TC-CALC-014` | Same format as the traceability matrix. |
| Clause | `<STANDARD>-<NUMBER>` | `ISO-14971-7.3` | Same format as the traceability matrix `clause_refs`. |

## Enumerations

### Severity

Five-level scale. The matching numeric weights used for matrix derivation are documented
in `risk-matrix.md`.

| Value | Description |
|-------|-------------|
| `negligible` | No injury or only superficial. Recoverable without intervention. |
| `minor` | Reversible injury or temporary discomfort; no professional medical attention required. |
| `serious` | Reversible injury requiring professional medical attention. |
| `critical` | Irreversible injury or one requiring intervention to prevent permanent impairment. |
| `catastrophic` | Death or permanent loss of major function. |

### Probability

Five-level qualitative scale. The matching numeric weights are in `risk-matrix.md`.

| Value | Frequency description |
|-------|----------------------|
| `improbable` | Unlikely to occur in the device lifetime even across the installed base. |
| `remote` | Possible but unlikely; expected fewer than once per device lifetime. |
| `occasional` | Likely to occur occasionally during the device lifetime. |
| `probable` | Will probably occur several times during the device lifetime. |
| `frequent` | Likely to occur often during normal use. |

### Risk Level

Three-level acceptance scale. Derived from severity x probability via the matrix.

| Value | Action required (per ISO-14971-6 and 7.3) |
|-------|-------------------------------------------|
| `acceptable` | No further risk reduction required. Document the acceptance rationale in `notes`. |
| `ALARP` | Risk reduced as low as reasonably practicable. Acceptable if the cost of further reduction is grossly disproportionate to the benefit; rationale in `notes`. |
| `unacceptable` | Further risk-control measures required, or the design must be reconsidered. `evaluate --ci` exits non-zero on any record in this band. |

### Status Field

| Status | Meaning |
|--------|---------|
| `draft` | Record is being drafted; control measures may be incomplete. `validate` warns rather than errors. |
| `controlled` | Control measures applied and verified; residual risk evaluated. The normal post-mitigation state. |
| `accepted` | Residual risk is `unacceptable` per the matrix but accepted on benefit-risk grounds (ISO-14971-7.4). `notes` MUST contain `benefit-risk:` followed by a justification reference. |
| `transferred` | Risk-control responsibility transferred to another component or process (e.g. labelling). `notes` should name the receiving component. |
| `deferred` | Mitigation deferred to a later release. `notes` MUST contain `deferred:` followed by a justification. `evaluate --ci` warns rather than errors on these records. |

The skill writes one of the five values; consumers must not invent new ones.

## Validation Rules

The `validate` subcommand applies the rules below. The `clause_id` column is the ISO 14971
clause whose violation the rule maps to; the skill body must include this id in finding
messages so an auditor can pivot to the standard.

| Rule | Severity | Clause id |
|------|----------|-----------|
| Every record has at least one `control_measures[]` entry. | error | ISO-14971-7.1 |
| Every `control_measures[]` entry has at least one `verification[]` entry. | error | ISO-14971-7.2 |
| Every `requirements[]` entry resolves to a doc_id in `manifest.yaml`. | error | ISO-14971-7.2 |
| Every `clause_refs[]` value resolves to a clause id in `compliance/iso-14971.md`. | error | (skill-internal cross-reference) |
| Every `severity` and `probability` value is in the declared enum. | error | ISO-14971-5.5 |
| `initial_risk` and `residual_risk` match the matrix derivation from their inputs. | error | ISO-14971-6 (initial), ISO-14971-7.3 (residual) |
| Records in status `accepted` carry `benefit-risk:` in `notes`. | error | ISO-14971-7.4 |
| Records in status `deferred` carry `deferred:` in `notes`. | warning | (project-policy convention) |
| Records whose only control_measures `priority` is `3` carry a justification in `notes`. | warning | ISO-14971-7.1 (priority hierarchy) |
| `residual_severity > severity` (controls cannot worsen severity in this schema). | error | ISO-14971-7.5 |
| Records in status `controlled` whose `residual_risk` is `unacceptable`. | error | ISO-14971-7.3 |

In default mode (`/risk-control validate`), errors print to stdout and the skill exits 0.
In `--ci` mode (`/risk-control validate --ci`), errors cause a non-zero exit and warnings
print without affecting the exit code (matching `traceability`'s convention).

## Evaluation Output

The `evaluate` subcommand prints a Markdown table to stdout, sorted by `residual_risk`
descending then `id` ascending:

```markdown
## Risk Evaluation

| id    | hazard                                | initial      | residual     | status     | finding |
|-------|---------------------------------------|--------------|--------------|------------|---------|
| R-01  | Unintended therapy delivery           | unacceptable | ALARP        | controlled | -       |
| R-04  | Sensor disconnect during therapy      | unacceptable | unacceptable | controlled | ISO-14971-7.3: residual_risk unacceptable; further controls required |
| H-12  | Power loss mid-procedure              | ALARP        | acceptable   | controlled | -       |

Summary: evaluated 3 records; unacceptable=1 ALARP=1 acceptable=1
```

The `finding` column is empty (`-`) when no rule applies and contains the clause id plus a
short message when the residual risk is `unacceptable` or another rule fires.

## Example: Minimal Risk File

A small but complete `risk-file.yaml` for a project with a single risk record:

```yaml
_meta:
  schema: "1.0.0"
  generated: "2026-05-04T12:34:56Z"
  generator: "risk-control"
  matrix_source: "default"
  records: 1

records:
  - id: R-01
    hazard: "Unintended therapy delivery"
    sequence_of_events: |
      1. Operator selects pre-set protocol
      2. Embedded validation skips a parameter check
      3. Device delivers therapy outside approved range
    harm: "Patient receives over-dose"
    severity: critical
    probability: occasional
    initial_risk: unacceptable
    control_measures:
      - id: CM-01
        description: "Add range validation in firmware"
        priority: 1
        verification:
          - "TC-CALC-014: range-validation unit test"
          - "TC-INT-007: end-to-end protocol limits test"
    residual_severity: critical
    residual_probability: remote
    residual_risk: ALARP
    requirements:
      - SRS-CALC-014
    clause_refs:
      - ISO-14971-5.4
      - ISO-14971-7.1
      - ISO-14971-7.3
    status: controlled
    notes: ""
```

## Example: Accepted Residual With Benefit-Risk

```yaml
- id: R-07
  hazard: "Diagnostic device false negative for rare condition"
  sequence_of_events: |
    1. Patient with rare condition X presents
    2. Algorithm trained on insufficient X cases returns negative
    3. Clinician proceeds without escalation
  harm: "Delayed diagnosis"
  severity: critical
  probability: remote
  initial_risk: ALARP
  control_measures:
    - id: CM-01
      description: "Continuous-learning surveillance with quarterly retraining"
      priority: 2
      verification:
        - "TC-ML-022: post-deployment surveillance threshold test"
  residual_severity: critical
  residual_probability: remote
  residual_risk: ALARP
  requirements:
    - SRS-ML-022
  clause_refs:
    - ISO-14971-7.3
    - ISO-14971-7.4
  status: accepted
  notes: "benefit-risk: see docs/risk-management/benefit-risk-r07.md; clinical benefit for non-X patients outweighs the residual risk for X subgroup per literature [Kim 2024]."
```

The `notes:` `benefit-risk:` prefix is the contract that `validate` checks for `accepted`
records.

## Example: Deferred Mitigation

```yaml
- id: R-09
  hazard: "USB drive removed during firmware update"
  sequence_of_events: |
    1. Operator initiates firmware update from removable media
    2. Operator removes the media before the write completes
    3. Device boots into corrupted state requiring service
  harm: "Device unavailable until serviced; no patient harm"
  severity: minor
  probability: occasional
  initial_risk: ALARP
  control_measures:
    - id: CM-01
      description: "A/B partition with rollback on boot failure"
      priority: 1
      verification:
        - "TC-FW-031: power-loss-during-update test"
  residual_severity: minor
  residual_probability: remote
  residual_risk: acceptable
  requirements: []
  clause_refs:
    - ISO-14971-7.3
  status: deferred
  notes: "deferred: A/B partition implementation blocked by bootloader rework #602; revisit at v1.2 milestone."
```

## Schema Evolution

Schema changes are versioned via `_meta.schema`. The current major is `1`. Bump rules:

| Change | Bump |
|--------|------|
| Add a new optional field | Minor -- `1.0.0` -> `1.1.0`. Older consumers ignore the field. |
| Add a new allowed `status` value | Minor -- existing records unchanged; older consumers warn rather than fail on unknown statuses. |
| Add a new severity / probability level | Major -- breaks the matrix derivation contract; consumers must guard. |
| Rename or remove a field | Major -- `1.x.x` -> `2.0.0`. Consumers must guard on the major. |
| Change the meaning of an existing field | Major. |
| Adjust validation rules | Minor (consumers re-read this file). |

The `risk-control` skill writes the matching schema version into `_meta.schema` on every
write and refuses to operate on a file whose major does not match.
