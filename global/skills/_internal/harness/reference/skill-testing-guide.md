## 1. Testing Framework Overview

Skill quality validation combines **qualitative** and **quantitative** evaluation.

| Evaluation type | Method | Suited for |
|----------------|--------|-----------|
| **Qualitative** | User reviews the output directly | Subjective quality: writing style, design, creative work |
| **Quantitative** | Assertion-based automated grading | Objectively verifiable: file generation, data extraction, code generation |

Core loop: **write -> test run -> evaluate -> improve -> re-test**

## 2. Test Prompt Writing

### Principles

Test prompts must be **specific, natural sentences that a real user would actually type**. Abstract or artificial prompts have low test value.

### Bad Examples

```
"Process the PDF"
"Extract the data"
"Generate a chart"
```

### Good Examples

```
"In my downloads folder, there's 'Q4_Sales_Final_v2.xlsx'. Add a profit margin
(%) column using column C (revenue) and column D (cost). Then sort by profit
margin in descending order."
```

```
"Extract the table on page 3 from this PDF and convert it to CSV. The table
header spans 2 rows -- the first row is the category, the second row has
the actual column names."
```

### Prompt Diversity

- Mix **formal / casual** tones
- Mix **explicit / implicit** intent (directly stating file format vs requiring inference from context)
- Mix **simple / complex** tasks
- Include some abbreviations, typos, and casual expressions

### Coverage

Start with 2-3 prompts, designed to cover:
- 1 core use case
- 1 edge case
- (Optional) 1 complex/compound task

## 3. Execution Testing: With-skill vs Baseline

### 3-1. Comparison Execution Structure

For each test prompt, spawn two sub-agents **simultaneously**:

**With-skill execution:**
```
Prompt: "{test prompt}"
Skill path: {skill path}
Output path: _workspace/iteration-N/eval-{id}/with_skill/outputs/
```

**Baseline execution:**
```
Prompt: "{test prompt}"  (identical)
Skill: none
Output path: _workspace/iteration-N/eval-{id}/without_skill/outputs/
```

### 3-2. Baseline Selection

| Situation | Baseline |
|-----------|----------|
| Creating a new skill | Run the same prompt without any skill |
| Improving an existing skill | Previous version of the skill (preserve a snapshot) |

### 3-3. Timing Data Capture

Save `total_tokens` and `duration_ms` **immediately** from the sub-agent completion notification. This data is only accessible at notification time and cannot be recovered later.

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

## 4. Quantitative Evaluation: Assertion-based Grading

### 4-1. Writing Assertions

When output is objectively verifiable, define assertions for automated grading.

**Good assertions:**
- Objectively determinable as true/false
- Descriptive names that make it clear what is being checked from the result alone
- Verify the skill's core value

**Bad assertions:**
- Always pass regardless of skill presence (e.g., "output exists")
- Require subjective judgment (e.g., "well written")

### 4-2. Programmatic Verification

When assertions can be verified with code, write scripts. This is faster and more reliable than visual inspection and reusable across iterations.

### 4-3. Non-discriminating Assertion Warning

Assertions that "pass 100% for both configurations" do not measure the skill's differential value. When you find such assertions, remove them or replace with more challenging ones.

### 4-4. Grading Result Schema

```json
{
  "expectations": [
    {
      "text": "Profit margin column added",
      "passed": true,
      "evidence": "Column E contains 'profit_margin_pct'"
    },
    {
      "text": "Sorted by profit margin descending",
      "passed": false,
      "evidence": "Original order maintained, no sorting applied"
    }
  ],
  "summary": {
    "passed": 1,
    "failed": 1,
    "total": 2,
    "pass_rate": 0.50
  }
}
```

## 5. Specialized Agent Utilization

Leveraging specialized agents in the testing/evaluation process improves quality.

### 5-1. Grader

Performs assertion-based grading and extracts verifiable claims from outputs for cross-verification.

**Role:**
- Pass/fail determination for each assertion + evidence citation
- Extract factual claims from outputs and verify them
- Provide feedback on the eval itself (suggest when assertions are too easy or ambiguous)

### 5-2. Comparator (Blind Comparator)

Anonymizes two outputs as A/B and evaluates quality without knowing which used the skill.

**When to use:** When you need a rigorous answer to "Is the new version really better?" Can be skipped during routine iterative improvement.

**Judgment criteria:**
- Content: accuracy, completeness
- Structure: organization, formatting, usability
- Overall score

### 5-3. Analyzer

Analyzes statistical patterns from benchmark data:
- Non-discriminating assertions (both configurations pass -> no differential value)
- High-variance evals (results vary significantly between runs -> instable)
- Time/token tradeoffs (skill improves quality but also increases cost)

## 6. Iterative Improvement Loop

### 6-1. Collecting Feedback

Show outputs to the user and collect feedback. Empty feedback is interpreted as "no issues".

### 6-2. Improvement Principles

1. **Generalize the feedback** -- narrow fixes for specific test examples are overfitting. Fix at the principle level.
2. **Remove what doesn't pull its weight** -- read transcripts: if the skill is directing the agent to do unproductive work, delete that section.
3. **Explain the why** -- even if the user's feedback is brief, understand why it matters and reflect that understanding in the skill.
4. **Bundle repetitive work** -- if every test run generates the same helper script, pre-include it in `scripts/`.

### 6-3. Iteration Procedure

```
1. Modify the skill
2. Re-run all test cases in a new iteration-N+1/ directory
3. Present results to the user (compare with previous iteration)
4. Collect feedback
5. Modify again -> repeat
```

**Termination conditions:**
- User is satisfied
- All feedback is empty (no issues with any output)
- No further meaningful improvements possible

### 6-4. Draft -> Review Pattern

When modifying skills, write a draft first, then **re-read with fresh eyes** and improve. Don't try to write perfectly in one pass; go through draft-review cycles.

## 7. Description Trigger Verification

### 7-1. Trigger Eval Query Writing

Write 20 eval queries -- 10 should-trigger + 10 should-NOT-trigger.

**Query quality criteria:**
- Specific, natural sentences a real user would type
- Include concrete details: file paths, personal context, column names, company names, etc.
- Mix of length, tone, and format
- Focus on **boundary cases (edge cases)** rather than clear-cut answers

**Should-trigger queries (8-10):**
- Same intent expressed in diverse phrasings (formal/casual)
- Skill/file type not explicitly mentioned but clearly needed
- Non-mainstream use cases
- Cases competing with other skills but where this skill should win

**Should-NOT-trigger queries (8-10):**
- **Near-misses are key** -- similar keywords but a different tool/skill is appropriate
- Obviously unrelated queries ("write a Fibonacci function") have no test value
- Adjacent domains, ambiguous phrasing, keyword overlap but different context

### 7-2. Existing Skill Collision Check

Verify that the new skill's description does not overlap with existing skills' trigger areas:

1. Collect descriptions from the existing skill list.
2. Confirm the new skill's should-trigger queries don't mistrigger existing skills.
3. If collisions are found, clarify boundary conditions in the description.

### 7-3. Automated Optimization (Optional Advanced Feature)

When description optimization is needed:

1. Split 20 eval queries into Train (60%) / Test (40%).
2. Measure trigger accuracy with the current description.
3. Analyze failure cases and generate an improved description.
4. Select the best description based on Test set (not Train set -- to prevent overfitting).
5. Maximum 5 iterations.

> This process uses `claude -p` for automation. Token cost is high, so run it in the final stage after the skill is sufficiently stable.

## 8. Workspace Structure

Directory structure for systematically managing test/evaluation results:

```
{skill-name}-workspace/
  iteration-1/
    eval-descriptive-name-1/
      eval_metadata.json
      with_skill/
        outputs/
        timing.json
        grading.json
      without_skill/
        outputs/
        timing.json
        grading.json
    eval-descriptive-name-2/
      ...
    benchmark.json
  iteration-2/
    ...
  evals/
    evals.json
```

**Rules:**
- Use **descriptive names** for eval directories, not numbers (e.g., `eval-multi-page-table-extraction`)
- Each iteration is preserved in an independent directory (never overwrite previous iterations)
- `_workspace/` is never deleted -- for post-verification and audit trails
